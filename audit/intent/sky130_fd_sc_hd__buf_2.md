# T4.1 — Intent doc: `sky130_fd_sc_hd__buf_2` (foundry-supplied, used as MWL driver)

## What this cell IS electrically

Foundry standard-cell buffer from the SKY130 hd library. Used by rekolektion as the **MWL row driver** in CIM macros: instantiated once per row, takes `MWL_EN[r]` as input and drives `mwl_<r>` output to the row of supercells.

This cell is **not** generated by rekolektion. We consume the PDK-supplied GDS and extracted SPICE.

**Port list** (from the cached extract): `A VGND VNB VPB VPWR X` — 6 ports. Standard buffer convention: A=input, X=output, VPWR/VGND=supply, VPB/VNB=body bias.

## Source

- Foundry GDS: from SKY130 hd library; copied/referenced at build time.
- Cached extracted body: `src/rekolektion/peripherals/cells/extracted_subckt/sky130_fd_sc_hd__buf_2.subckt.sp` (11 lines).

## How rekolektion uses it

- `cim_spice_generator.py:_write_supercell` and the per-row instance loop:
  ```
  Xmwl_<r> MWL_EN[r] VGND VGND VPWR VPWR mwl_<r> sky130_fd_sc_hd__buf_2
  ```
  Port mapping: A=`MWL_EN[r]`, VGND=VGND, VNB=`VGND` (forced), VPB=`VPWR` (forced), VPWR=VPWR, X=`mwl_<r>`.
- 64 instances per macro (one per row), tiled into `cim_mwl_driver_col_64`.
- Layout-side: the buf_2 GDS reference is instantiated by `cim_mwl_driver_col_64`.

## Diff vs intent

| Item | Foundry-supplied | Rekolektion's instance binding | Comment |
|------|--------------------|----------------------------------|---------|
| Port list | `A VGND VNB VPB VPWR X` | `MWL_EN[r] VGND VGND VPWR VPWR mwl_<r>` | OK — VNB tied to VGND (substrate), VPB tied to VPWR (n-well). Standard. |
| Body bias | VPB / VNB are explicit ports, no in-cell well taps | Rekolektion ties VNB→VGND and VPB→VPWR via reference netlist instance line, but **has not added physical N-tap inside the buf_2 GDS** | The buf_2 GDS is foundry-fixed; rekolektion cannot modify it. The N-tap to VPWR must come from **outside** the buf_2 cell — i.e., from a peripheral N-tap in `cim_mwl_driver_col_64` or from row-stripping. **Subject to T4.4 flood-fill.** |

## Severity flags

- **N-A** for cell-internal-topology audit — foundry-supplied, not rekolektion's.
- **P1 candidate — N-well biasing for buf_2 instances.** Same concern as for the bitcell array NWELL: if the buf_2 instances' n-wells are not physically tapped to a VPWR rail outside the cell, body bias is undefined per row driver. T4.4 main-session result was that the bitcell array has all NWELLs floating; need to confirm whether `cim_mwl_driver_col_64` provides N-tap geometry around the 64 buf_2 instances. The label survey shows `cim_mwl_driver_col_64` has 130 unique label texts but only `VPWR: 1, VGND: 1` at the col level — not 64 of each — suggesting the rail is shared and broken into peripheral pieces. Need T2-style trace to confirm.
