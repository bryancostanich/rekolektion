### T1.1 finding — P0 reference-pipeline bug

The production-macro LVS is **partially self-referential**: at LVS time, Magic extracts `BitcellArray`, `PrechargeRow`, `ColumnMuxRow` from the current GDS, the result is embedded into the reference SPICE, and then *that* reference is compared against the same layout's flat extraction. Any layout error in those sub-cells gets baked into the reference and the comparison can't see it.

**This is exactly the failure pattern the WL_BOT/WL_TOP bug exploited.** F11+F13's "33793=33793 net match in `sram_array`" is therefore weaker evidence than it appeared — it shows the layout is internally consistent with itself, not that it matches a hand-written intent.

**Recommendation:** the BitcellArray reference must be hand-written (or a one-shot extraction snapshot committed to git, with a regeneration audit before tapeout). Same for PrechargeRow and ColumnMuxRow. Otherwise we have no independent ground truth at the `sram_array` sub-circuit level.

T2.1 flood-fill is the test that closes this gap.

## T1.5 — Port-alignment scope

Source: `_align_ref_ports` in `scripts/run_lvs_cim.py:48-73` and `scripts/run_lvs_production.py:129-160`.

Both implementations:
- Take the extracted SPICE's `.subckt cell_name` port list
- Rewrite the **reference SPICE's** `.subckt cell_name` line to match
- Skip continuation lines (`+`) under the rewritten subckt
- Touch **only the line matching `.subckt cell_name`** (the macro top), never any sub-cell `.subckt` definition

**Verdict: PASS.** Confirmed by reading both functions; the loop matches `stripped.startswith(f".subckt {cell_name}")` and only modifies that single subckt's port list. Sub-cell .subckt blocks are passed through verbatim.

| `src/rekolektion/macro/liberty_generator.py` | **Analytical formulas only** — array-size + SKY130 device/interconnect parameters; no SPICE simulation, no extracted netlist | **NOT MEASURED** — see header comment lines 3-4: "analytically-computed timing values" |
| `src/rekolektion/macro/cim_liberty_generator.py` | **Analytical formulas only** — explicitly acknowledges "A future SPICE-characterisation pass should replace these analytical numbers with sim-extracted values" | **NOT MEASURED** — see header comment lines 11-13 |

### T1.7 finding — P1 timing-data unverified

The Liberty timing data shipping in `.lib` files is computed from analytical models, not measured from SPICE simulation. Per `trust_audit.md:122-124` this is the failure pattern (timing data not derived from extracted SPICE = unverified). The generators acknowledge this as a known limitation in their docstrings.

**Severity: P1 functional.** The macros will integrate against analytical timing; if real silicon timing differs from analytical estimates by >20% (plausible for `MBL_OUT` which depends on cap-charge sharing, see `cim_liberty_generator.py:11`), downstream timing closure on the SoC will see surprises. **Not silicon-breaking** — the chips will still work, just slower or faster than the .lib claims.

**Recommendation:** declare Liberty as analytical-estimate-only in tapeout signoff; plan Liberty re-characterisation (already filed as task #24 / F10) before any timing-critical integration.
