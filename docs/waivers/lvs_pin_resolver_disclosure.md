# Tapeout disclosure waiver — LVS top-level pin-resolver artifact

**Macros affected:** `sram_activation_bank`, `sram_weight_bank_small`, `cim_sram_a/b/c/d` (all 6).
**Severity classification:** P2 (verification artifact only — no silicon-correctness implication).
**Status:** Accepted for tapeout pending designer sign-off on this disclosure.

---

## What netgen / Magic actually report

Every macro in this codebase exhibits one of two related Magic ext2spice port-promotion artifacts at the macro top:

### CIM (`cim_sram_a/b/c/d`) — `MWL_EN[r]` + `MBL_<c>` reconciliation

The reference SPICE generated by `cim_spice_generator.generate_cim_reference_spice` declares per-row `MWL_EN[r]` ports (the row-enable signals for the compute path). Inside the layout, the labels exist correctly: 64 (or 256) `MWL_EN[r]` text labels on layer 67/5 (li1 label) at the boundary of the `cim_mwl_driver_col_<rows>` cell. **A standalone Magic extract of the driver-column cell finds and promotes all of them as ports.** But Magic's hierarchical extract (`port makeall` recursive) at the macro top does *not* propagate those sub-cell ports up to the top-level subckt.

Symmetrically, the Magic extract finds 64 (or 256) per-column `MBL_<c>` strap labels at the macro boundary that the current SPICE generator does not declare (these are legacy strap labels still present in the layout).

### Production (`sram_weight_bank_small`, `sram_activation_bank`) — `addr[0..6]` disconnect

The reference SPICE declares an 8 or 9-bit address bus (`addr[0..8]` / `addr[0..7]`). Magic's hierarchical extract finds the boundary `addr[i]` labels but does *not* connect them through the row_decoder hierarchy at the macro top — the address rails remain "disconnected pins" at the top level even though they connect physically.

For both macros the mismatch is exactly the bus that's used by the row decoder (`addr[0..6]`). Verified by reading `comp.out` for both production macros (2026-05-03, task #105).

## Why this is a tooling artifact and not a silicon defect

Independent flood-fill on the extracted netlist (`audit/flood_fill_2026-05-03.md`, task #110) confirms:
- **Per-row/col electrical nets fan out across the full array.** wl/mwl/bl/br/mbl per-row/col labels each appear in 64+ (or 256+) device-instance lines inside `cim_array_<v>_<rxc>` — i.e. each row's WL touches every cell in that row.
- **VPWR / VGND merge correctly** across all 4096 (CIM 64×64) / 16384 (CIM 256×64 + production) supercells + periphery.
- **Zero auto-NWELL fragments** in CIM (T5.2-A's resolution holding); production has one well-cluster covered by the existing `nwell_bias_disclosure.md` waiver.
- **Issue #7-class per-cell drain floats** are not present (Phase 2 drain bridge holding).

The full silicon connectivity is verified at the netlist level. The only thing Magic's hierarchical port-promotion fails to do is propagate certain labeled boundary rails up through the cell hierarchy to the macro-top subckt's port list.

This was already established by the prior session in commit `b09c441` (F12): "Magic's hierarchical port-promotion refuses to merge a child cell's interior addr[i] rail with the parent's feeder, **regardless of .pin shape placement**." A flat-extraction fix was tried and reverted in `a97f56f` because flat extraction hides issue #7-class per-cell drain floats — so we live with the hierarchical-extraction artifact.

## Risk profile

| Risk | Magnitude | Mitigation |
|------|-----------|------------|
| LVS at the macro top reports "Top level cell failed pin matching" | Cosmetic on the LVS report; sub-circuit comparisons all match net-perfect | `_align_ref_ports` (`scripts/run_lvs_cim.py:54`, `scripts/run_lvs_production.py:129`) reconciles the .subckt port lists.  As of 2026-05-03 it's hardened with explicit per-script allow-lists (`MWL_EN[r]` / `MBL_<c>` for CIM; `dec_out_<N>` for production) and fails loud on any unexpected drift. |
| SoC-level integration trusts the LEF and/or hand-written reference SPICE rather than the post-aligner LVS report | None — the LEF and reference are the authoritative interface specs, both correctly enumerate every external pin | Manual port verification at SoC integration is the standard cross-check; LVS is a supporting tool, not the primary boundary spec |
| Future macro variant introduces a new port pattern not in the allow-list | Aligner aborts the LVS run with a precise diagnostic naming the unexpected port + the file:line where the allow-list lives | Update the allow-list in the corresponding script after verifying the new port is real; the failure message points at `audit/hack_inventory.md` for the audit handle |

## Aligner allow-list reference

For audit traceability, the aligner allow-lists at the time of this disclosure are:

**CIM** (`scripts/run_lvs_cim.py`):
- Allow drops matching `^MWL_EN\[\d+\]$` (Magic does not promote per-row driver enables through hierarchy)
- Allow adds matching `^MBL_\d+$` (legacy macro-top column-strap labels Magic finds in the layout)

**Production** (`scripts/run_lvs_production.py`):
- Allow drops: empty pattern (any drop fails the run)
- Allow adds matching `^dec_out_\d+$` (Magic over-promotes internal row-decoder li1 outputs from `row_decoder_<tag>` to the macro top)

**Per-cell substitution** (`scripts/extract_cim_subckts.py:_PORT_LIST`):
- Allows the same Magic-limitation pattern at the peripheral cell level (cim_mbl_precharge, cim_mbl_sense, sky130_fd_sc_hd__buf_2). Documented inline.

Each allow-list is paired with a `_check_*_drift` function that aborts the run on any port outside the documented patterns. Silent absorption of new drift is not possible.

## Scope of this waiver

- Applies to all 6 macros listed above on the **CI2605 shuttle** (or any shuttle taping out from this codebase before either Magic's port-promotion is fixed, or a two-pass hierarchical+flat extraction reconciliation is implemented).
- Does NOT waive any other category of finding. The 5 P0 silicon defects identified in the trust audit (T1.1-A, T2.1-CIM-A, T4.2-CIM-A, T5.2-A, original T4.4-A) have all been independently resolved with traceable evidence and are not covered by this waiver.
- Does NOT waive the bitcell + periphery N-well biasing — see `nwell_bias_disclosure.md` for that scope.
- Does NOT waive the analytical-Liberty-not-SPICE-measured concern — see `liberty_timing_analytical.md`.
